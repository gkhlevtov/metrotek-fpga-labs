## Детальное описание ошибок

### Баг №1: Ошибка логики флага `empty_o`

**Описание:**
После подачи сигнала `srst_i` и установления `usedw_o` в 0, флаг `empty_o` остается в состоянии 0, что противоречит состоянию пустой очереди.
При этом после записи и чтения одного слова данных флаг корректно устанавливается в 1.

**Ожидаемое поведение:** `empty_o=1`, `usedw_o=0`

**Фактическое поведение:** `empty_o=0`, `usedw_o=0`.

**Сообщения в transcript:**

```
# ** Error: [SCB] @35: EMPTY mismatch!   Got: 0, Exp: 1
#    Time: 35 ps  Scope: lifo_pkg.Scoreboard...
```

**Время первого воспроизведения:** `T = 35 ps`

---

### Баг №2: Отсутствие защиты от Underflow

**Описание:**

После синхронного сброса (`srst_i=1`) LIFO находится в пустом состоянии (`usedw_o=0`).
При подаче команды чтения (`rdreq_i=1`) на пустую очередь ожидается, что состояние не изменится.
Фактически DUT декрементирует внутренний указатель без проверки на пустоту, что приводит к целочисленному переполнению счётчика `usedw_o` (wrap-around): значение уходит в `511 = 2^(AWIDTH+1) - 1`.

**Ожидаемое поведение:** `empty_o=1`, `usedw_o=0` - состояние не изменяется.

**Фактическое поведение:** `empty_o=0`, `usedw_o=511`.

**Сообщения в transcript:**

```
[SCB] @95: Read from empty LIFO - underflow protection triggered
** Error: [SCB] @95: EMPTY mismatch!   Got: 0, Exp: 1
   Time: 95 ps  Scope: lifo_pkg.Scoreboard...
** Error: [SCB] @95: USEDW mismatch! Got: 511, Exp: 0
   Time: 95 ps  Scope: lifo_pkg.Scoreboard...
```

**Время первого воспроизведения:** `T = 95 ps`

### Баг №3: Ошибка логики флага `almost_full_o`

**Описание:**

Флаг `almost_full_o` должен быть равен `1` при выполнении условия `usedw_o >= ALMOST_FULL_VALUE`.
При `ALMOST_FULL_VALUE=2` флаг обязан подняться начиная с `usedw_o=2`.
Фактически DUT использует инвертированный порог: флаг поднимается при `usedw_o >= 2^AWIDTH - ALMOST_FULL_VALUE = 254`, то есть параметр ошибочно интерпретируется как количество свободных слов, а не занятых.

**Ожидаемое поведение:** `almost_full_o=1` при `usedw_o >= 2`.

**Фактическое поведение:** `almost_full_o=0` при `usedw_o=2..253`, `almost_full_o=1` при `usedw_o >= 254`.

**Сообщения в transcript:**

```
** Error: [SCB] @105: ALMOST_FULL mismatch!  Got: 1, Exp: 0 (usedw=0, ALMOST_FULL=2)
   Time: 105 ps  Scope: lifo_pkg.Scoreboard...

   ...

** Error: [SCB] @165: ALMOST_FULL mismatch!  Got: 0, Exp: 1 (usedw=2, ALMOST_FULL=2)
   Time: 165 ps  Scope: lifo_pkg.Scoreboard...
   
   ...
```

**Время первого воспроизведения:** `T = 105 ps`

### Баг №4: Отсутствие защиты от Overflow

**Описание:**

После заполнения LIFO до максимальной ёмкости (`usedw_o=256`, `full_o=1`) при подаче дополнительных команд записи (`wrreq_i=1`) ожидается, что состояние не изменится.
Фактически DUT продолжает инкрементировать внутренний указатель без проверки на заполненность, `usedw_o` превышает значение `256`.

**Ожидаемое поведение:** `usedw_o=256` - состояние не изменяется.

**Фактическое поведение:** `usedw_o` растёт выше 256.

**Сообщения в transcript:**

```
[SCB] @2715: Write to full LIFO - overflow protection triggered
** Error: [SCB] @2715: USEDW mismatch! Got: 257, Exp: 256
   Time: 2715 ps  Scope: lifo_pkg.Scoreboard...
```

**Время первого воспроизведения:** `T = 2715 ps`

### Баг №5: Ошибка логики флага `almost_empty_o`

**Описание:**

При чтении из очереди элементов флаг `almost_empty_o` должен устанавливаться в 1, сигнализируя о том, что очереднь наполнена <=`ALMOST_EMPTY_VALUE` словами.
Фактически флаг устанавливается не при достижении значения `ALMOST_EMPTY_VALUE`, а только при достижении значения, меньшего на 1.

**Ожидаемое поведение:** `almost_empty_o` примет значение 1, когда в очереди останется <=`ALMOST_EMPTY_VALUE = 2` элемента.

**Фактическое поведение:** `almost_empty_o` принимает значение 1, когда в очереди остается <=`ALMOST_EMPTY_VALUE-1 = 1` элемент.

**Сообщения в transcript:**

```
** Error: [SCB] @2845: ALMOST_EMPTY mismatch! Got: 0, Exp: 1 (usedw=2, ALMOST_EMPTY=2)
   Time: 2845 ps  Scope: lifo_pkg.Scoreboard...
```

**Время первого воспроизведения:** `T = 2845 ps`

### Баг №6: Ошибка одновременной записи и чтения (`wrreq_i=1, rdreq_i=1`)

**Описание:**

При одновременной подаче `wrreq_i=1` и `rdreq_i=1`, когда в очереди имеется хотя бы одно слово, ожидается, что:

* происходит чтение верхушки стека (`q_o` = текущий top),
* одновременно записывается новое слово (`data_i`),
* размер стека (`usedw_o`) не изменяется.

Фактически в ситуации, когда в очереди лежит только 1 слово, DUT игнорирует `wrreq_i` и данные с `data_i`, постоянно подавая на `q_o` исходное слово.

При этом для очередей, наполненных 2 словами и более, одновременная запись и чтение работают корректно.

**Ожидаемое поведение:** `usedw_o` остаётся неизменным; `q_o` отражает вновь записанные данные.

**Фактическое поведение:** `usedw_o` не изменяется; на выходе `q_o` остается самое первое записанное слово; все последующие данные, подаваемые на `data_i` во время одновременной операции, игнорируются.

**Сообщения в transcript:**

```
** Error: [SCB] @2905: DATA MISMATCH! Got: 2b8a, Exp: 984e
   Time: 2905 ps  Scope: lifo_pkg.Scoreboard...
** Error: [SCB] @2915: DATA MISMATCH! Got: 2b8a, Exp: ea12
   Time: 2915 ps  Scope: lifo_pkg.Scoreboard...
   
   ...
```

**Время первого воспроизведения:** `T = 2905 ps`
