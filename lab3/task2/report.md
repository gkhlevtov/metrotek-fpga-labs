## Детальное описание ошибок

### Баг №1: Некорректное значение `empty_o` для пакета размером ровно в одно выходное слово

**Описание:**
При передаче пакета, который состоит ровно из **N** входных beat’ов (N = DATA_OUT_W / DATA_IN_W = 4) и при этом последний beat имеет `empty_i = 0` (все байты значащие), ожидается, что выходное слово будет полностью заполнено и `empty_o = 0`.
Однако DUT выдаёт `empty_o = 31`, что соответствует всего одному значащему байту. Ошибка проявляется только в случае, когда последний входной beat содержит `empty_i = 0`, при других значениях `empty_i` выходной `empty_o` вычисляется верно.

**Ожидаемое поведение:** `empty_o = 0` (все 32 байта выходного слова значащие).

**Фактическое поведение:** `empty_o = 31` (только 1 значащий байт).

**Сообщение в transcript:**

```
** Error: [SCB] @65 beat #1: EMPTY mismatch! Got=31 Exp=0
   Time: 65 ps  Scope: ast_width_extender_pkg.Scoreboard.Scoreboard__1.check_beat File: scoreboard.sv Line: 141
```

**Время первого воспроизведения:** `T = 65 ps`

---

### Баг №2: Отсутствие `endofpacket_o` для пакета из одного входного beat’а

**Описание:**
Когда пакет состоит всего из одного входного beat’а (длина 1), то есть `startofpacket_i` и `endofpacket_i` установлены одновременно, DUT не формирует сигнал `endofpacket_o` на выходе. Выходной beat выставляется с `sop=1`, но `eop=0`, при этом `valid=1`. Для пакетов с большим количеством beat данная ошибка не наблюдается, модуль корректно выставляет `eop=1`.

**Ожидаемое поведение:** для пакета длиной 1 должен быть сформирован один выходной beat с `sop=1`, `eop=1` и `empty_o`, рассчитанным на основе входного `empty_i`.

**Фактическое поведение:** выходной beat имеет `sop=1`, `eop=0`, а `empty_o` соответствует количеству значащих байт (например, `empty=29` при `empty_i=3`).

**Сообщение в transcript:**

```
** Error: [SCB] @855 beat #16: Expected EOP not received!
   Time: 855 ps  Scope: ast_width_extender_pkg.Scoreboard.Scoreboard__1.check_beat File: scoreboard.sv Line: 136
```

**Время первого воспроизведения:** `T = 855 ps`

---

**Примечание:**

- Также стоит отметить в DUT выставление сигнала `startofpacket=1` на несколько тактов, например, на временной диаграмме в тайминги @25-@65. Так как при последующих тактах удержания `startofpacket` сигнал `valid` не устанавливается, то на взаимодействие с модулем это не влияет.
